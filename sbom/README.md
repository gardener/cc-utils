# BOM Documents â€” Storage Convention

SBOM (Software Bill of Materials) and CBOM (Cryptography Bill of Materials) documents are
generated for every OCI image resource and stored in two ways: as **OCI referrer manifests**
(primary, for direct OCI consumers) and as **OCM resources** (for consumers working through
component descriptors).

## SBOM

Generated by [syft](https://github.com/anchore/syft) in two formats.

### OCI referrers

Each image gets two SBOM referrer manifests stored in the same OCI repository, linked to the
image by digest via the [OCI 1.1 referrers API][oci-referrers]:

| Format | `artifactType` |
|--------|----------------|
| SPDX 2.3 | `application/spdx+json` |
| CycloneDX 1.6 | `application/vnd.cyclonedx+json` |

Retrieve with [`oras`][oras]:

```sh
oras discover --artifact-type application/spdx+json <image-ref>
oras blob fetch <repo>@<referrer-digest> --output sbom.spdx.json
```

Or directly via the registry API:

```
GET /v2/<repo>/referrers/<digest>?artifactType=application%2Fspdx%2Bjson
```

The SBOM document is the single layer of the referrer manifest
(`manifest.layers[0].digest`).

The referrer manifest carries two annotations:
- `org.opencontainers.image.created` â€” scan timestamp (ISO 8601)
- `gardener.cloud/sbom/tool-version` â€” syft version used (if available)

### OCM resources

In OCM component descriptors the SBOM documents appear as additional resources alongside
the image resource they describe.  Both share the image resource's `name` and `version`;
the `extraIdentity` field distinguishes them:

| Field | Value |
|-------|-------|
| `sbom-format` | `spdx-2.3` or `cyclonedx-1.6` |
| `version` | same as the source image resource |
| platform fields | `os`, `architecture` (present when the source image is platform-specific) |

The resource `type` matches the SBOM media type above.

Labels on each SBOM resource:

| Label | Content |
|-------|---------|
| `gardener.cloud/sbom/source-image` | original image reference |
| `gardener.cloud/sbom/source-image-digest` | digest of the scanned image manifest |
| `gardener.cloud/sbom` | `{data-source: {tool, tool-version}, format}` |

Access is either `localBlob` (SBOM inlined into the component descriptor blob store) or
`ociRegistry` pointing at the referrer manifest digest.

## CBOM

Generated by [cbomkit-theia](https://github.com/IBM/cbomkit-theia) (IBM), which takes the
CycloneDX SBOM produced by syft as `--bom` input and enriches it with cryptographic asset
findings (certificates, keys, secrets, OpenSSL/Java security config). This avoids a second
full image pull.

### OCI referrer

Each image gets one CBOM referrer manifest:

| Format | `artifactType` |
|--------|----------------|
| CycloneDX 1.6 | `application/vnd.cyclonedx+json;profile=cbom` |

The `;profile=cbom` parameter distinguishes CBOM referrers from CycloneDX SBOM referrers
when querying the referrers API.

Retrieve:

```sh
oras discover --artifact-type 'application/vnd.cyclonedx+json;profile=cbom' <image-ref>
oras blob fetch <repo>@<referrer-digest> --output cbom.cdx.json
```

The referrer manifest carries:
- `org.opencontainers.image.created` â€” scan timestamp (ISO 8601)
- `gardener.cloud/cbom/tool-version` â€” cbomkit-theia version (if available)

### OCM resource

One additional resource per image, distinguished by `extraIdentity.cbom-format`:

| Field | Value |
|-------|-------|
| `cbom-format` | `cyclonedx-1.6` |
| `version` | same as the source image resource |
| platform fields | `os`, `architecture` (when platform-specific) |

Labels:

| Label | Content |
|-------|---------|
| `gardener.cloud/cbom/source-image` | original image reference |
| `gardener.cloud/cbom/source-image-digest` | digest of the scanned image manifest |
| `gardener.cloud/cbom` | `{data-source: {tool, tool-version}, format}` |
